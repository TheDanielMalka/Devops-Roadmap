Issue #52: Docker Concepts & Architecture (COMPLETED)
- Problem Docker solves: "works on my machine" - packages app + dependencies in one container
- Container vs VM: Container shares host Kernel (lightweight, MBs, seconds to start) vs VM runs full OS (heavy, GBs, minutes)
- Docker Engine 3 parts:
  - Client: CLI commands in terminal (docker run, docker ps)
  - Daemon (dockerd): runs in background, does all the work (pull, run, manage)
  - Registry (Docker Hub): online storage for ready-made images (nginx, python, postgres)
- Flow: You type command → Client sends to Daemon → Daemon checks locally → pulls from Registry if missing → runs container
- Docker Desktop: installs Client + Daemon + GUI together, uses WSL2 for Daemon
- Image = blueprint (read-only template), Container = running instance (can run multiple from same image)
- Installed Docker on WSL2, verified with docker run hello-world

Issue #53: Running & Inspecting Containers (COMPLETED)
- docker run: pulls image if missing, creates and starts container
- Detached mode (-d): runs container in background, frees terminal
- docker ps: shows running containers. ps -a shows all including stopped
- docker logs: view container output, first thing to check when debugging
- docker exec -it <name> bash: enter container shell, temporary access for debugging
- docker stop: graceful shutdown. docker start: restart stopped container
- docker rm: delete stopped container. rm -f: force stop + delete
- --rm flag: auto-delete container when it stops
- Port mapping (-p 8080:80): maps host port to container port, required for external access
- Multiple containers can use same internal port, each gets unique host port
- docker inspect: full container details (state, network, config, ports)
- Container isolation: closed by default, -p opens specific ports to outside

Issue #56: Docker Volumes & Data Persistence (COMPLETED)
- Containers lose all data when removed - proved by creating file, deleting container, new container = gone
- Named Volume: Docker manages folder, portable (same command works everywhere), data survives container deletion
- Bind Mount: maps host folder to container, syncs both ways in real time, good for dev but not portable (path must exist)
- tmpfs: stored in RAM only, deleted when container stops, for temporary/sensitive data
- Difference: Named Volume = Docker manages, Bind Mount = you manage. Both persist data, only portability differs
- Practical: Postgres with pgdata volume, created table + inserted data, deleted container, new container = data survived
- Volume is local only - other machines start with empty volume, not meant for sharing data between machines

Issue #57: Writing Dockerfiles (COMPLETED)
- Dockerfile = recipe for building custom image, line by line instructions
- FROM: base image (python:3.11-slim). Always use specific version not latest
- WORKDIR: sets working directory inside image, creates if not exists
- COPY: copies files from project folder into image. Only sees files in Dockerfile's directory
- Two COPYs pattern: copy requirements first, install, then copy code - saves rebuild time with caching
- RUN: executes commands during build (pip install, mkdir). Each RUN = new layer. Combine with &&
- ENV: sets environment variables baked into image. Like -e flag but permanent
- EXPOSE: documentation only, does not open port. -p flag actually opens ports
- CMD: default command when container starts, can be overridden. ENTRYPOINT: fixed, cannot override
- docker build -t <name> . builds image. -t names image, dot = current directory
- -t = name for image (build), --name = name for container (run)
- Dockerized TaskManager: built image, ran container, added volume for data persistence
- .dockerignore works like .gitignore - excludes files from image Opus 4.6

- docker-compose.yml structure: services (containers), volumes, networks. Compose auto-creates network for all services

Issue #83: AWS VPC - Networking Fundamentals (COMPLETED)
- VPC = private virtual network in the cloud
- CIDR 10.0.0.0/16 defines address space (65,536 addresses)
- Public subnet: servers with external IP, internet accessible
- Private subnet: no external IP, internal only
- Internet Gateway: translates internal to external addresses, connects VPC to internet
- Route Tables: direct traffic - public subnet routes to IGW, private stays internal
- Security Groups: firewall per server, blocks all inbound by default
- NACLs: firewall per subnet, stateless (doesn't remember connections)
- Practical: Created custom "arena-vpc" with public + private subnet, IGW, route tables
